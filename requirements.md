# 要件定義書：推し活グッズ BtoB ECサイト

> 最終更新日: 2026-04-20
> バージョン: 1.1.0

---

## 1. プロジェクト概要

### 1.1 背景・目的
推し活グッズ（オリジナルデザイン・データ入稿を伴う商品に特化）をライセンス事業者向けに販売するBtoB ECサイトを構築する。

#### 背景
かざスキ事業の展開を通じて、ライセンス事業者とのコネクションが増加している。この関係性を活かし、推し活グッズの横展開によって事業者あたりの売上を最大化したい。

#### 課題
従来のグッズ業界では、営業担当が事業者に専属でつき、個別に対応するモデルが一般的である。しかしこのモデルには以下の課題がある：

- **提供側の非効率:** 事業者数の増加に比例して人員が必要になり、スケールしない
- **事業者側の非効率:** 都度の問い合わせ・やりとりが発生し、発注までのリードタイムが長い

一方で、専属営業モデルが業界に根強く残っているのは、事業者にとって「商品選定の相談」「入稿データの確認」「柔軟な対応」といった価値があるからである。

#### 本サービスの目的
専属営業モデルが提供してきた価値（商品の探しやすさ、入稿サポート、確認フローの安心感）を、ECサイトのUXと仕組みで代替する。Amazonのように全てがシステマティックに動くECサイトではなく、裏側に人がいるがわかるような安心感を演出できるECサイト。これにより、人を介さずとも事業者が迷わず・効率的に発注できるセルフサーブ型の受注チャネルを構築する。

### 1.2 成功基準（KGI/KPI）
| 指標 | 目標値 | 備考 |
|------|--------|------|
| 登録事業者数 | 数百社 | 中長期目標 |
| その他KPI | TBD | 売上目標、CVR等は事業計画に基づき別途設定 |

### 1.3 ターゲットユーザー
- **購入者:** ライセンス事業者（推し活グッズの企画・販売を行う法人）のMD部署の担当者等
- **管理者:** 自社の運営担当者（主に鬼柳、八木）

### 1.4 デザイン方針
- **参考サイト:** Casetify（カスタム商品EC）、無印良品（シンプルなデザイン）
- **UX方針:** Amazon等の一般的なECサイトと同様の操作感。違和感のないUXを重視。
- **アイコンライブラリ:** Lucide（`lucide-react`パッケージ）
- **価格表記:** @XX円（税込）〜（ロット数10,000個時の単価）
- **告知バー:** ヘッダー最上部に入稿サポートの訴求を常時表示（PC版のみ）

## 2. 用語集

本書で使用する用語の定義。認識ズレを防ぐため、プロジェクト全体でこの定義に統一する。

| 用語 | 定義 |
|------|------|
| **アクター** | |
| 購入者 | ECサイトで商品を注文するライセンス事業者 |
| 管理者 | ECサイトを運営する自社の担当者 |
| **商品** | |
| オプション | 商品ごとに設定されるバリエーション項目（サイズ、色等）。項目自体が商品ごとに異なる |
| ロット | 注文数量の単位 |
| 価格テーブル | ロット数に応じた単価の一覧。商品ごとに設定 |
| 入稿テンプレート | 商品ごとに用意された、入稿データ作成用のテンプレートファイル |
| **注文フロー** | |
| 注文 | 購入者がカート内の全商品について入稿を完了し、注文を確定すること。商品選択→カート追加→入稿→注文確定の流れで行われる |
| 入稿 | 購入者がデザインデータをアップロードすること |
| 入稿スロット | 商品ごとに定義された入稿データの枠。1スロット＝1ファイル。管理者がスロットのラベル（表面デザイン、裏面デザイン等）、必須/任意、表示条件（常時 or 特定オプション選択時）を設定 |
| 再入稿 | 差し戻し後に購入者がデザインデータを再アップロードすること |
| 承認 | 管理者が入稿データを確認し、問題ないと判断すること |
| 差し戻し | 管理者が入稿データに問題を発見し、修正を依頼すること |
| 決済リンク | 管理者が承認後に購入者へメールで送信する支払い用URL。有効期限あり |
| リピート注文 | 過去の注文と同じ商品・同じ入稿データで再注文すること |
| **サンプル** | |
| サンプル制作 | オリジナルデザインの商品を少量で製造する注文。通常注文と同じく入稿が必要。有料。カート内で数量変更可能（個単位） |
| 色校正 | 購入者が入稿したデザインデータを元に試作し、色味を確認する工程。注文時のオプション選択 |

## 4. 業務フロー

### 4.1 会員登録フロー

```
サイト訪問 → アカウント作成（Supabase Auth） → お客様情報入力 → 登録完了 → 購入可能
```

| ステップ | アクター | 詳細 |
|----------|----------|------|
| 1. サイト訪問 | 購入者 | ECサイトにアクセス（商品閲覧は未ログインでも可能） |
| 2. アカウント作成 | 購入者 | Supabase Authで認証（ソーシャルログイン: Google, Apple, Facebook, X or マジックリンク） |
| 3. お客様情報入力 | 購入者 | 初回ログイン時にお客様情報（会社名（任意）・お名前・電話番号・住所）を入力。「あとで入力する」でスキップ可能 |
| 4. 登録完了 | システム | 登録完了メールを送信（NTF-001）。注文・入稿が可能になる |

### 4.2 基本注文フロー（正常系）

```
商品選択 → カートに追加（複数商品可）→ 各商品ごとに入稿 → 全商品入稿完了 → 注文確定
→ [管理者] 入稿確認 → 承認 → 決済リンク送信 → [購入者] 支払い → 製造 → 発送 → 完了
```

| ステップ | アクター | 詳細 | ステータス |
|----------|----------|------|------------|
| 1. 商品選択 | 購入者 | 商品を選択し、オプション（サイズ・色等）と数量を決定 | — |
| 2. 見積もり確認 | 購入者 | 数量・オプションに連動した価格・納期目安を確認。必要に応じて見積書PDFをダウンロード | — |
| 3. カートに追加 | 購入者 | 商品をカートに追加。複数商品をカートに入れてまとめて注文可能 | — |
| 4. 入稿 | 購入者 | カートページから商品ごとに入稿ページへ遷移し、デザインデータをアップロード（複数ファイル対応、1ファイル200MB上限）。ライセンス同意チェック | — |
| 5. 注文確定 | 購入者 | 全商品の入稿が完了した状態で、注文内容を確認し確定する | 入稿受付 |
| 6. 受付通知 | システム | 購入者に注文・入稿受付メール送信（NTF-002）。管理者に新規注文通知（NTF-008） | — |
| 7. 入稿確認 | 管理者 | 入稿データを確認（数営業日） | 入稿確認中 |
| 8. 承認 | 管理者 | データに問題がなければ承認 | 承認済み |
| 9. 決済案内送信 | 管理者 | 購入者が選択した支払い方法に応じた決済案内をメールで送信（NTF-005）。クレカ: Stripe決済リンク / 銀行振込: 振込先情報＋請求書 / 請求書払い: 請求書 | 決済待ち |
| 10. 支払い | 購入者 | 決済リンクから支払い（有効期限あり） | 入金確認 |
| 11. 入金確認通知 | システム | 購入者に入金確認メール送信（NTF-006） | — |
| 12. 製造 | 管理者 | 製造開始 | 製造中 |
| 13. 発送 | 管理者 | 納品書を同梱して発送。発送完了メール送信（NTF-007） | 発送済み |
| 14. 完了 | — | 購入者に届いた | 完了 |

### 4.3 サンプル制作フロー

サンプル制作は通常注文と同じフロー（4.2）で処理する。違いは以下のみ：

- **入り口:** 商品詳細ページの「サンプル制作をカートに追加」ボタンからカートに追加
- **注文種別:** カート内で「サンプル」として表示・管理される
- **数量:** カート内で数量変更可能（個単位。CRT-007参照）
- **以降のフロー:** カートに追加後の入稿・注文確定・確認・決済・製造・発送は通常注文と共通

### 4.4 見積もりフロー

```
商品選択 → オプション・数量入力 → リアルタイム価格表示 → 見積書PDFダウンロード
```

| ステップ | アクター | 詳細 |
|----------|----------|------|
| 1. 商品選択 | 購入者 | 商品詳細ページでオプション・数量を入力 |
| 2. 価格確認 | 購入者 | 数量・オプションに連動してリアルタイムで価格・納期が表示される |
| 3. 見積書DL | 購入者 | 必要に応じて見積書をPDFでダウンロード。社内稟議等に利用 |

> **備考:** 見積もりは注文フローの一部としても機能するが、見積書DLだけで注文に進まないケースもある。

### 4.5 問い合わせフロー

```
商品ページ → 問い合わせフォーム入力 → 送信 → 管理者対応 → メール回答
```

| ステップ | アクター | 詳細 |
|----------|----------|------|
| 1. フォーム入力 | 購入者 | 商品ページから問い合わせフォームを開き、内容を入力 |
| 2. 送信 | 購入者 | フォームを送信 |
| 3. 受付通知 | システム | 購入者に問い合わせ受付メールを送信。管理者にメールで通知 |
| 4. 回答 | 管理者 | メールで購入者に回答 |

### 4.6 例外フロー

| 例外 | 発生タイミング | 対応 |
|------|----------------|------|
| 入稿データに不備 | 入稿確認時 | 管理者がメールで差し戻し連絡（NTF-004）→ 購入者がサイト上から再入稿 → 再度確認 |
| 決済リンク期限切れ | 決済待ち | 管理者が新しい決済リンクを再送信 |
| 決済失敗（カードエラー等） | 支払い時 | 購入者にエラーを表示し、別の決済方法での再決済を案内 |
| アップロードエラー | 入稿時 | エラーメッセージを表示し、再アップロードを促す |
| 入稿データの形式不正 | 入稿時 | アップロード時にバリデーションし、許可形式を案内 |
| ファイルサイズ超過 | 入稿時 | エラーメッセージで上限（200MB）を案内 |
| 最小ロット未満の注文 | 数量入力時 | 注文画面でバリデーションし、最小ロット数を案内 |
| 初期不良 | 納品後 | 自社起因の初期不良のみ返品対応。問い合わせ経由で受付 |
| キャンセル依頼 | 注文後 | 原則不可。例外対応は問い合わせ経由 |

### 4.7 注文方式
- **カート方式** で複数商品をまとめて注文可能
- カート内の各商品ごとに入稿を行い、全商品の入稿が完了した時点で注文確定に進める

> **意思決定の経緯:** 当初は1商品ずつの注文を想定していたが、社内協議の結果、事業者の利便性を考慮してカート方式を初期から採用する方針に変更。

## 5. 機能要件

### 優先度の定義（MoSCoW法）
| 優先度 | 意味 |
|--------|------|
| **Must** | 初期リリースに必須。これがないとサービスが成立しない |
| **Should** | 初期リリースに強く推奨。可能な限り含める |
| **Could** | あると良いが、なくても運用可能 |

### 5.1 購入者向け機能

#### 会員機能

| ID | 機能 | 詳細 | 優先度 | 受入基準 |
|----|------|------|--------|----------|
| USR-001 | 会員登録 | 会員登録すれば誰でも購入可能 | Must | 必要項目を入力し、登録完了メールを受信できる |
| USR-002 | 認証 | Supabase Auth。ソーシャルログイン（Google、Apple、Facebook、X）＋マジックリンク（メール） | Must | 各ソーシャルログインおよびマジックリンクでログインできる |
| USR-003 | 登録項目 | メールアドレス（必須・ソーシャルログインで取得できた場合は自動入力）、会社名（任意）、お名前、電話番号、住所。初回ログイン時にスキップ可能だが、入稿・注文確定時に未登録ならC-11（会員情報編集ページ）に遷移して入力を求める | Must | 全項目の入力・バリデーションが機能する。未登録でも商品閲覧・カート追加は可能 |
| USR-004 | マイページ | 会員情報の閲覧・編集 | Must | 登録情報を変更し、保存できる |
| USR-005 | 退会 | 問い合わせ経由で管理者が対応 | Could | 問い合わせフォームから退会依頼を送信できる |

#### 商品閲覧

| ID | 機能 | 詳細 | 優先度 | 受入基準 |
|----|------|------|--------|----------|
| PRD-001 | 商品一覧 | カテゴリ別表示 | Must | カテゴリを選択し、該当商品の一覧が表示される |
| PRD-002 | 商品検索 | キーワード検索 | Must | キーワードを入力し、商品名・説明文から検索結果が表示される |
| PRD-003 | カテゴリ絞り込み | カテゴリ別フィルタリング | Must | カテゴリを選択し、該当商品のみに絞り込める |
| PRD-004 | 商品詳細ページ | 商品画像（複数枚）、説明、価格テーブル、納期目安、入稿テンプレートDL | Must | 全項目が正しく表示される |
| PRD-005 | 画像切替 | 選択したオプションに応じて商品画像が切り替わる | Should | オプションを変更すると対応する画像に切り替わる |
| PRD-006 | 関連商品表示 | 商品ページに関連商品を表示 | Should | 商品詳細ページ下部に関連商品が表示される |
| PRD-007 | 即時見積もり | 数量・オプション選択に連動してリアルタイムで価格表示 | Must | 数量・オプションを変更すると即座に価格が更新される |
| PRD-008 | 納期目安表示 | 商品・数量に応じた納期目安を表示 | Must | 数量を入力すると納期目安が表示される |

#### 注文・入稿

| ID | 機能 | 詳細 | 優先度 | 受入基準 |
|----|------|------|--------|----------|
| ORD-001 | オプション選択 | 商品ごとに異なるオプション項目（サイズ、色等） | Must | 商品に設定されたオプション項目が表示され、選択できる |
| ORD-002 | 最小ロット | 商品ごとに最低注文数量を設定 | Must | 最小ロット未満の数量ではエラーが表示される |
| ORD-003 | 数量別価格 | ロット数に応じた価格テーブル（商品ごと） | Must | 商品詳細ページに数量別の価格テーブルが表示される |
| ORD-004 | データ入稿 | 入稿ページでスロット別にデザインデータをアップロード（1スロット＝1ファイル）。スロットは商品ごとに管理者が定義。特定オプション選択時のみ出現するスロットもある | Must | 商品に設定されたスロットが表示され、指定形式のファイルをアップロードできる |
| ORD-005 | 複数ファイル対応 | 1注文につき複数ファイルのアップロード可能 | Must | 複数ファイルを選択・アップロードできる |
| ORD-006 | ファイルサイズ上限 | 1ファイルあたり200MB | Must | 200MB超のファイルはエラーメッセージが表示される |
| ORD-007 | データ形式指定 | 商品ごとに入稿データの形式を指定 | Must | 許可されていない形式はエラーメッセージが表示される |
| ORD-008 | 入稿テンプレート | 商品ページ・入稿ページからスロットごとのテンプレートファイルをダウンロード可能。オプション選択に連動して表示が変わる | Must | 該当するテンプレートファイルをダウンロードできる |
| ORD-009 | 再入稿 | 差し戻し後、サイト上から再入稿可能 | Must | 差し戻された注文に対してファイルを再アップロードできる |
| ORD-010 | サンプル制作 | 商品詳細ページからサンプル制作をカートに追加可能（有料・入稿必要・数量変更可能） | Must | サンプル制作としてカートに追加でき、通常注文と同じフローで注文できる |
| ORD-011 | 色校正オプション | 注文時に色校正ありを選択可能。色校正のやりとり自体はメールベースで対応（システム上のフロー分岐なし） | Should | 色校正オプションを選択でき、注文内容に反映される |
| ORD-012 | ライセンス同意 | 注文確認ページ（C-06）で同意チェックボックス（同意記録を保存）。入稿ページではなく注文確定の直前で1回のみ | Must | チェックなしでは注文確定できない。同意日時が記録される |

#### カート

| ID | 機能 | 詳細 | 優先度 | 受入基準 |
|----|------|------|--------|----------|
| CRT-001 | カートに追加 | 商品詳細ページからカートに商品を追加（オプション・数量を含む）。ログイン不要。未ログイン時はブラウザにローカル保存し、ログイン時にサーバーと同期 | Must | 未ログインでも商品がカートに追加され、ヘッダーのカートアイコンに数量が反映される |
| CRT-002 | カート一覧表示 | カート内の商品一覧を表示。商品名・オプション・数量・価格・入稿状態を表示 | Must | カート内の全商品と合計金額が正しく表示される |

## 10. 制約・前提条件

### 10.1 前提条件
- 新規構築（既存データの移行は不要）
- 受注生産モデル（在庫管理は不要）
- 免税事業者（インボイス制度非対応）

### 10.2 制約
- 予算・納期・技術スタックはエンジニアと相談の上決定
- デザインカンプは別途作成（デザイナー or エンジニアと相談）

### 10.3 別途準備が必要なもの
| 項目 | 担当 | 期限 |
|------|------|------|
| 利用規約の文面 | 自社 | リリース前 |
| プライバシーポリシーの文面 | 自社 | リリース前 |
| 特商法表記の文面 | 自社 | リリース前 |
| 送料無料の閾値（N円）の決定 | 自社 | リリース前 |
| 決済リンクの有効期限の決定 | 自社 | リリース前 |
| 独自ドメインの取得 | 自社 | リリース前 |
| 商品データ（画像・説明・価格等）の準備 | 自社 | リリース前 |

### 10.4 返品・キャンセルポリシー
- **返品:** 自社起因の初期不良のみ対応
- **キャンセル:** 原則不可
